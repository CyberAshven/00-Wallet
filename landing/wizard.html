1. Overview

WizardConnect is a BCH HD wallet connection protocol created by RiftenLabs. It allows dapps and wallets to communicate securely via Nostr relays — exchange xpubs for address derivation and request transaction signatures.

00 Protocol implements BOTH sides:
  • WALLET side — External dapps (Cauldron DEX, etc.) scan our QR code to connect
  • DAPP side — 00 Protocol connects to external wallets (Electron Cash, Cashonize, etc.)

2. Architecture

Wallet Side (00 Wallet serves external dapps)

Dapp scans QR -> Connects to relay -> Sends DappReady -> 00 Wallet sends xpubs -> Dapp derives addresses

Dapp Side (00 Protocol connects to external wallets)

User pastes wiz:// URI -> 00 Protocol connects -> Sends DappReady -> Wallet sends xpubs -> 00 Protocol derives addresses

6. HD Paths & xpubs

WizardConnect sends xpubs for each named path. The dapp derives addresses locally — no private keys are transmitted.

// Path index mapping (wizardconnect.js, line 41)
const WIZ_PATH_INDEX = {
  receive:       0,   // m/44'/145'/0'/0    — standard receive addresses
  change:        1,   // m/44'/145'/0'/1    — change addresses
  defi:          7,   // m/44'/145'/0'/7    — DeFi interactions (RiftenLabs standard)
  stealth_spend: 0,   // m/352'/145'/0'/0'  — stealth spend (BIP352, HARDENED, separate tree)
  stealth_scan:  1,   // m/352'/145'/0'/1'  — stealth scan  (BIP352, HARDENED, separate tree)
  rpa:          47,   // m/47'/145'/0'      — RPA (BIP47 paycode, separate tree)
};

// HD tree structure:
m/44'/145'/0'                    (account node — xpriv stays in wallet)
  +-- 0   receive  (non-hardened)  -> xpub sent to dapp
  +-- 1   change   (non-hardened)  -> xpub sent to dapp
  +-- 7   defi     (non-hardened)  -> xpub sent to dapp

m/352'/145'/0'  <- BIP352 tree (stealth, SEPARATE from BIP44)
  +-- 0'  spend   (HARDENED)     -> xpub sent via WizardConnect
  |    +-- 0     spend key        -> dapp derives locally
  +-- 1'  scan    (HARDENED)     -> xpub sent via WizardConnect
       +-- 0     scan key         -> dapp derives locally

m/47'/145'/0'   <- BIP47 tree (RPA/paycode, SEPARATE from BIP44)
  +-- /0          notification key
  +-- /0-n        ECDH payment keys
BIP352 isolation: Stealth keys live in a completely separate tree (m/352'/145'/0'). Even if the BIP44 account xpub (m/44'/145'/0') leaks, stealth keys are unreachable. Scan and spend are behind independent hardened gates — compromising the scan xpub gives zero access to the spend key. Follows the BIP352 (Silent Payments) key derivation structure.
PR status: Adding "stealth" and "rpa" to WizardConnect PathName is approved by Dagur Valberg (WizardConnect creator). PR pending.

xpub derivation (Wallet side)

// _deriveChildXpub (wizardconnect.js, line 392)
// Wallet derives child xpub from account node for each path

// For non-hardened paths (receive, change, defi):
const child = _bip32ChildPub(acctPub, acctChain, childIndex);

// For hardened path (stealth):
// Wallet uses xpriv internally: _bip32Child(acctPriv, acctChain, 2, true)
// Then extracts the public key + chain code to build the xpub
// This xpub is sent in the handshake

// xpub format: base58check encoded (version 0x0488B21E)
// 78 bytes: version(4) + depth(1) + fingerprint(4) + index(4) + chain(32) + key(33)

Address derivation (Dapp side)

// derivePubkey (wizardconnect.js, line 630)
// Dapp receives xpub, derives child pubkeys locally

const dapp = new WizardConnect.DappManager('My Dapp');
dapp.connect('wiz://...');

dapp.onConnect((walletName, walletIcon, paths) => {
  // Derive receive address at index 0
  const pub0 = dapp.derivePubkey('receive', 0);  // 33-byte compressed pubkey

  // Derive stealth scan/spend pubkeys (if stealth path available)
  const scanPub = dapp.getStealthScanPub();   // 33 bytes
  const spendPub = dapp.getStealthSpendPub(); // 33 bytes
});

7. Stealth Integration

Stealth addresses work seamlessly with WizardConnect. The key insight (from community review):

  • Stealth SEND can work from ANY wallet via WizardConnect. The dapp (00 Protocol) does the ECDH + address derivation and builds a normal P2PKH transaction. The connected wallet just signs it — it doesn't even know it's a stealth payment.
  • Stealth RECEIVE requires the scan private key + scanning logic (ECDH, Nostr DM, chain scan). This must live in the wallet itself. WizardConnect is just a transport — it cannot scan on behalf of the wallet.
// Stealth send via WizardConnect (dapp side):

// 1. Dapp receives stealth xpubs from wallet handshake
//    stealth_scan  = xpub at m/352'/145'/0'/1'  (scan branch, hardened)
//    stealth_spend = xpub at m/352'/145'/0'/0'  (spend branch, hardened)
const scanPub  = deriveChild(stealthScanXpub, 0);   // m/352'/145'/0'/1'/0
const spendPub = deriveChild(stealthSpendXpub, 0);  // m/352'/145'/0'/0'/0

// 2. Dapp generates paycode
const paycode = 'stealth:' + hex(scanPub) + hex(spendPub);

// 3. To send stealth TO another user's paycode:
//    Dapp does ECDH locally, builds TX with stealth output
//    Then requests wallet to sign via sign_transaction_request
const signedTx = await dapp.requestSign(unsignedTx, inputPaths);

// The wallet signs a normal P2PKH — it's stealth-unaware!
// Stealth receive via WizardConnect (wallet side):
// NOT possible via WizardConnect alone.
// The wallet must implement stealth scanning natively.
// See stealth.html for scanning implementation.

8. Transaction Signing

Dapp requests signature

// requestSign (wizardconnect.js, line 651)
const signedTx = await dapp.requestSign(
  {
    raw: unsignedTxHex,    // unsigned TX hex
    outputs: [             // human-readable output descriptions
      { address: 'bitcoincash:qr...', value: 50000, label: 'Payment' },
    ],
  },
  [
    [0, 'receive', 3],     // input 0: sign with receive path, index 3
    [1, 'change', 0],      // input 1: sign with change path, index 0
  ]
);
// Returns: signed TX hex (or throws on reject/timeout)
// Timeout: 5 minutes

Wallet approves/rejects

// Wallet side — user prompt (wizardconnect.js, line 425)
wallet.onSignRequest((req) => {
  // Show confirmation UI to user
  if (userApproves) {
    const signedHex = signTransaction(req.transaction, req.inputPaths);
    wallet.approveSign(req.sequence, signedHex);
  } else {
    wallet.rejectSign(req.sequence, 'user_rejected');
  }
});

9. Class Reference

WizRelay Transport

Low-level Nostr relay connection. Handles WebSocket, NIP-04 encryption/decryption, auto-reconnect.
