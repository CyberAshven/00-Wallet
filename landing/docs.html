Introduction

00 Protocol is a decentralized ecosystem built on Bitcoin Cash. Zero central server, zero custody, zero borders. Every app runs directly in the browser, peer-to-peer.

All private keys stay in your browser or on your hardware wallet. No data is sent to third parties. BCH transactions are broadcast directly to the network via public relays.

No account. No server. No KYC. Your private key = your identity.

00 Chat

What it does — End-to-end encrypted messaging with information-theoretic security. Messages are split into two halves using a XOR one-time pad: one half goes on-chain via BCH OP_RETURN, the other half is sent over Nostr relays. Neither channel alone reveals anything — even with unlimited computing power.

Real-life example: Alice wants to message Bob. She types "Hello Bob". The app encrypts it, then XOR-splits the ciphertext into two random-looking blobs. Blob A is embedded in a BCH transaction to Bob's address. Blob B is published to Nostr relays. Bob's app finds both halves, XOR-merges them, decrypts → "Hello Bob". If the government seizes the blockchain data → they have random noise. If they wiretap Nostr → random noise. You need both to read anything.
// ── HOW IT WORKS ────────────────────────────────────────────── // // Alice Bob // │ │ // ├─ plaintext "Hello Bob" │ // ├─ ephemeral X25519 keypair │ // ├─ ECDH(eph_priv, bob_pub) → shared secret │ // ├─ AES-256-GCM(SHA256(shared), plaintext) → ciphertext │ // ├─ XOR split: [shard, pad] = xorSplit(ciphertext) │ // │ │ // ├──── BCH OP_RETURN ──── shard (chain half) ──────────► │ // │ 546 sats dust TX on-chain forever │ // │ │ // ├──── Nostr relay ────── pad (relay half) ──────────► │ // │ kind 21059 ephemeral 1h TTL │ // │ │ // │ xorMerge(shard, pad) │ // │ AES-GCM decrypt │ // │ → "Hello Bob" ✓ │

Step 1 — Ephemeral ECDH (forward secrecy)

Every single message generates a fresh X25519 keypair. Even if your long-term key is compromised, past messages remain safe.

// Real code from chat.html — per-message ECDH const ephPriv = crypto.getRandomValues(new Uint8Array(32)); const ephPub = x25519.getPublicKey(ephPriv); const shared = x25519.getSharedSecret(ephPriv, recipientPub); const innerKey = sha256(shared); // AES-256-GCM encryption (12-byte nonce, 128-bit auth tag) const nonce = crypto.getRandomValues(new Uint8Array(12)); const ct = await crypto.subtle.encrypt( { name: 'AES-GCM', iv: nonce, tagLength: 128 }, key, plaintext );

Step 2 — XOR one-time pad split

The ciphertext is XOR-split into two shares. Each share alone is pure random noise — Shannon-proven unbreakable.

// Real code — XOR split (information-theoretic security) function xorSplit(data) { const pad = crypto.getRandomValues(new Uint8Array(data.length)); const shard = new Uint8Array(data.length); for (let i = 0; i < data.length; i++) shard[i] = data[i] ^ pad[i]; // XOR: shard ⊕ pad = data return [shard, pad]; } function xorMerge(shard, pad) { const out = new Uint8Array(shard.length); for (let i = 0; i < shard.length; i++) out[i] = shard[i] ^ pad[i]; // reverse XOR to recover data return out; }

Step 3 — Dual-channel delivery

Each half gets its own AES-GCM envelope with a channel-specific key, then delivered via two independent networks.

// Real code — dual-channel key derivation const chainKey = sha256(concat(shared, "ccsh-chain")); // BCH on-chain half const relayKey = sha256(concat(shared, "ccsh-relay")); // Nostr relay half // Wrap each half with its own AES-GCM layer const chainBlob = aesWrap(shard, chainKey); // → BCH OP_RETURN const relayBlob = aesWrap(pad, relayKey); // → Nostr kind 21059

Step 4 — BCH transaction (chain half)

// CCSH v2 packet format MAGIC : 0x43435348 ("CCSH") VERSION : 0x02 MSG_TYPE : 0x10 = SPLIT_CHAIN | 0x11 = SPLIT_RELAY FLAGS : 0x01 (FLAG_SPLIT) MSG_ID : 16 bytes random SENDER_PUB : 32 bytes X25519 CHUNK_IDX : 2 bytes (big-endian) CHUNK_TOTAL : 2 bytes PAYLOAD : ephPub(32) || chainBlob // Transaction structure Output 0 : 546 sats → recipient P2PKH (dust, triggers inbox detection) Output 1 : OP_RETURN(CCSH packet) (0 sats, data carrier) Output 2 : change → sender (if ≥ 546 sats)

Step 5 — Inbox scanning

The recipient polls Fulcrum/Electrum every 10 seconds for new transactions. When a CCSH packet is detected on-chain, it waits for the matching Nostr relay half, XOR-merges, and decrypts.

00 Wallet

What it does — Non-custodial multi-chain wallet supporting BCH, BTC, and ETH. Generate keys, import seeds, connect a Ledger, or use WalletConnect. Full UTXO management, stealth addresses, and live portfolio tracking — all client-side.

Real-life example: You import your 12-word seed phrase. The app derives your BCH, BTC, and ETH keys from that single seed (BIP32/44). Your keys are encrypted with AES-256-GCM (200k PBKDF2 iterations) and stored locally. You see your balances, send coins, receive to stealth addresses — no server ever sees your keys.
// ── HOW IT WORKS ────────────────────────────────────────────── // // User Input Wallet Engine // │ │ // ├─ 12-word seed ──────────────► BIP39 → 64-byte master seed // │ │ // │ BIP32 Master Key // │ │ // │ 44'/145'/0' // │ ┌──┬──┬──┬──┬──┐ // │ 0/0 2/0 2/1 3/0 4/0 5/0 // │ │ │ │ │ │ │ // │ BCH Scan Spd BTC ETH XMR // │ │ // │ AES-256-GCM vault // │ PBKDF2 200k iter // │ │ // │ localStorage // │ (00wallet_vault) // │ │ // ├─ View balances ◄──── Fulcrum WebSocket (UTXO queries) // ├─ Send coins ────────► TX build → sign → broadcast // └─ Receive ◄────────── stealth addr (ECDH + point tweak)

Four access modes:

  • 01
    Hardware (Ledger) — WebHID connection. BIP44 address scan with gap limit 5, max 50 addresses. Private key never leaves the device. Signs via APDU commands.
  • 02
    Imported seed — BIP39 mnemonic (12+ words) or 128-char hex seed. PBKDF2-SHA512 (2,048 iterations, salt "mnemonic") → 64-byte master seed. Full HD derivation for all chains.
  • 03
    New key — 32 bytes from crypto.getRandomValues() (OS entropy: /dev/urandom, CryptGenRandom, SecRandomCopyBytes). Displayed once as 64-char hex for backup.
  • 04
    WalletConnect v2 — QR-based pairing. Mobile wallet signs transactions remotely. Session management via @walletconnect/sign-client.

00 Swap

What it does — Peer-to-peer atomic swaps between BCH and BTC. No intermediary, no exchange, no custody. Both parties lock funds in HTLC (Hash Time-Locked Contract) scripts on their respective chains, then claim each other's funds using preimage reveals.

Real-life example: Alice has BCH, Bob has BTC. Alice publishes an offer on the Nostr orderbook. Bob takes it. Alice locks her BCH in an HTLC for 24h. Bob locks his BTC in an HTLC for 12h. Bob claims Alice's BCH by revealing his preimage. Alice sees the revealed preimage on-chain, XOR-derives the other preimage, and claims Bob's BTC. If anything goes wrong — both can refund after their timelock expires.
// ── ATOMIC SWAP FLOW ───────────────────────────────────────── // // Taker (wants BTC) Maker (wants BCH) // │ │ // ├─ Generate P_bch (random 32 bytes) │ // ├─ P_btc = P_bch ⊕ SHA256(ECDH) │ // ├─ H_bch = SHA256(P_bch) │ // ├─ H_btc = SHA256(P_btc) │ // │ │ // ├──── Nostr 22241 ── {H_bch, H_btc} ──► │ // │ │ // │ ◄── Nostr 22242 ── {htlc_txid} ────┤ Lock BCH (24h) // │ maker locks BCH with H_bch │ // │ │ // ├─ Lock BTC (12h) with H_btc │ // ├─ Claim maker's BCH with P_bch ─────────┤ // │ │ // │ ◄── sees P_bch on-chain ───────────┤ // │ derives P_btc via XOR │ // │ claims taker's BTC ✓ │ // │ │ // └─ COMPLETE ✓ COMPLETE ✓

Preimage XOR decoration — Instead of using the same hash on both chains (which would link the swap), each chain gets a unique preimage derived via ECDH:

// Real code from swap.html — decorrelated preimages function crossChainPreimages(makerPub, takerPriv) { const P_bch = rand(32); // random preimage for BCH chain // ECDH shared secret between taker and maker const shared = secp256k1.getSharedSecret(takerPriv, '02' + makerPub); const mask = sha256(concat(utf8('swap-decorrelate'), shared.slice(1,33))); // XOR to derive BTC preimage — no hash correlation between chains const P_btc = new Uint8Array(32); for (let i = 0; i < 32; i++) P_btc[i] = P_bch[i] ^ mask[i]; return { P_bch, P_btc, H_bch: sha256(P_bch), H_btc: sha256(P_btc) }; }

HTLC script — Each HTLC is a standard Bitcoin conditional script with two paths:

// Real code — HTLC script construction (swap.html:997) function htlcScript(paymentHash32, recipientPub33, senderPub33, locktime) { // ┌── Claim path (preimage reveal) ──┐ // │ OP_IF │ // │ OP_SHA256 <hash> OP_EQUALVERIFY │ // │ <recipientPub> OP_CHECKSIG │ // ├── Refund path (timeout) ──────────┤ // │ OP_ELSE │ // │ <locktime> OP_CLTV OP_DROP │ // │ <senderPub> OP_CHECKSIG │ // │ OP_ENDIF │ // └───────────────────────────────────┘ }

State machine:

// Swap states with Nostr event kinds OFFERED kind 22240 Maker publishes offer to orderbook TAKEN kind 22241 Taker accepts, sends H_bch + H_btc MAKER_LOCKED kind 22242 Maker locks funds in HTLC (24h timeout) TAKER_LOCKED (on-chain) Taker locks funds in HTLC (12h timeout) COMPLETE kind 22243 Both sides claimed ✓ REFUNDED (on-chain) Timeout expired, funds recovered // Timeouts Maker : 144 blocks (~24 hours) ← longer, because maker locks first Taker : 72 blocks (~12 hours) ← shorter, claim window before maker

00 Onion

What it does — Onion-routed HTLC payments across the BCH network. Like Lightning Network but for Bitcoin Cash. Multi-hop encrypted routing where each relay node only knows the previous and next hop — never the full path or the sender/recipient.

Real-life example: Alice wants to pay Carol, but there's no direct channel. Alice routes through Bob. She wraps the payment instructions in 3 layers of encryption (one per hop). Bob can only peel his layer — he learns "forward X sats to Carol" but not that Alice is the sender. Carol peels the final layer, finds the preimage, and claims the payment. Each hop has its own unique HTLC — decorrelated hashes prevent linking.
// ── ONION ROUTING ──────────────────────────────────────────── // // Alice Bob (relay) Carol // │ │ │ // ├─ Build onion packet │ │ // │ Layer 3: [Bob's hop] ──encrypted──► │ // │ └ Layer 2: [Carol's hop] │ // │ └ Layer 1: [payment preimage] │ // │ │ │ // ├─ Lock HTLC ─── H₁ ──► Bob │ // │ ├─ Peel layer 3 │ // │ ├─ Lock HTLC ── H₂ ──► Carol // │ │ ├─ Peel final // │ │ ├─ Claim H₂ // │ ├─ Claim H₁ (fee ✓) │ // │ │ │ // │ H₁ ≠ H₂ (decorrelated — no hash linking)

Per-layer encryption — Each onion layer uses ephemeral secp256k1 ECDH + AES-256-GCM:

// Real code from onion.html:1169 — onion layer encryption async function onionLayer(payload, peelerPub) { const eph = rand(32); // ephemeral private key const ephPub = secp256k1.getPublicKey(eph, true); // compressed 33 bytes const shared = secp256k1.getSharedSecret(eph, peelerPub); const aesKey = sha256(shared.slice(1, 33)); // x-coordinate only const iv = rand(12); const ct = await aesGcmEncrypt(aesKey, iv, payload); return concat(ephPub, iv, ct); // [33 + 12 + variable] }

Hash decorrelation — Each hop gets a unique preimage derived from a master secret, preventing cross-hop linking:

// Real code — per-hop hash decorrelation (onion.html:1138) function deriveHopPreimage(masterSecret, hopIndex) { return sha256(concat(masterSecret, utf8('hop-' + hopIndex))); } function deriveHopHash(masterSecret, hopIndex) { return sha256(deriveHopPreimage(masterSecret, hopIndex)); } // Each hop sees a DIFFERENT hash — no correlation possible // Bob sees H₁ = SHA256(SHA256(master || "hop-0")) // Carol sees H₂ = SHA256(SHA256(master || "hop-1"))

Payload structure — Fixed 512-byte padded payload per hop, containing routing instructions:

// Onion payload per hop (512 bytes, 0x01-padded) Final hop : { amt, fee:0, cltv, hash, preimage, next:"final", stealth_spend, stealth_eph, sender_pub } Relay hop : { amt, fee, cltv, hash, preimage, next:"<next_pub>", next_hash, sender_pub, htlc_recipient_pub } HTLC : Same OP_IF/OP_ELSE script as 00 Swap Claim: <sig> <preimage> OP_1 <redeemScript> Refund: <sig> OP_0 <redeemScript> Constants : ONION_PAYLOAD_SIZE = 512 bytes DEFAULT_CLTV_DELTA = 6 blocks per hop RELAY_TTL = 300 seconds (5 min)

00 Vault

What it does — Invisible multisig vaults for Bitcoin Cash. Multiple cosigners control a shared wallet, but on-chain the vault address looks exactly like a regular single-signature P2PKH address. No script, no multisig indicator — invisible to blockchain analysis. This is the power of MuSig2: key aggregation produces a single 33-byte public key that hides the N-of-N structure.

Real-life example: Alice, Bob and Carol create a vault. Their 3 public keys are aggregated into one single public key via MuSig2. The vault address is a standard bitcoincash:q… — blockchain explorers see a normal address. Step 1 — Key aggregation (MuSig2)

All cosigner pubkeys are combined into a single aggregated public key. The vault address is simply HASH160(aggPub) — a standard P2PKH address.

Step 2 — Nonce exchange (via Nostr)

Before signing, each cosigner generates two random nonce points and broadcasts them encrypted via Nostr NIP-04 DMs.

// Real code — MuSig2 nonce generation (vault.html:741) function musigNonceGen() { const k1 = rand(32), k2 = rand(32); // secret nonces const R1 = G.multiply(k1Big).toRawBytes(true); // R₁ = k₁·G const R2 = G.multiply(k2Big).toRawBytes(true); // R₂ = k₂·G return { secnonce: { k1, k2 }, // kept secret pubnonce: concat(R1, R2) // 66 bytes → sent via Nostr }; }

Step 3 — Partial signatures & aggregation

Each cosigner computes a partial signature. When all partials arrive, they're summed into a single 64-byte Schnorr signature that verifies against the aggregated pubkey.

// Real code — partial signing (vault.html:782) // Each cosigner computes: sᵢ = (k₁ + b·k₂ + e·aᵢ·xᵢ) mod N function musigPartialSign(secnonce, myPriv, myCoeff, session) { const { b, e, negateR } = session; let k1 = modN(BigInt(secnonce.k1)); let k2 = modN(BigInt(secnonce.k2)); if (negateR) { k1 = modN(N - k1); k2 = modN(N - k2); } const si = modN(k1 + b * k2 + e * a * x); // partial sig return si; } // Aggregation: s = Σ sᵢ — final Schnorr signature (vault.html:795) function musigSigAgg(partialSigs, session) { let s = 0n; for (const ps of partialSigs) s = modN(s + ps); return concat(session.Rx, s); // 64 bytes: R.x(32) || s(32) } // Final scriptSig — looks identical to single-sig P2PKH: // <65: schnorrSig(64) + 0x41> <33: aggregated_pubkey> // A blockchain explorer literally cannot distinguish this from // a regular wallet sending a normal payment.

00 Joiner

What it does — Collaborative CoinJoin with stealth outputs. Multiple peers pool their UTXOs into a single transaction with shuffled outputs. Your coins go in with a traceable history — they come out to freshly derived stealth addresses that nobody can link back to you. Variable amounts (not fixed like traditional CoinJoin), onion-encrypted output shuffling, and peer coordination via Nostr relays.

ZERO TRACE Unlike every other CoinJoin / CashFusion implementation, 00 Joiner produces no OP_RETURN outputs. Traditional mixers embed protocol markers (OP_RETURN tags, version bytes, coordination data) that make mixed transactions trivially identifiable on-chain. 00 Joiner transactions are indistinguishable from regular multi-party transactions — no metadata, no fingerprint, no trace. Chain analysis tools cannot flag them as CoinJoin.
Real-life example: 5 users join a pool. Each contributes 1-5 UTXOs and receives 2-4 outputs at fresh stealth addresses. Stealth outputs — the secret weapon

Every output is sent to a freshly derived one-time stealth address. Each output gets its own ephemeral ECDH key, so even two outputs belonging to the same user are cryptographically unlinkable.

Outputs are wrapped in multi-layer onion encryption. Each peer peels one layer and performs a cryptographic shuffle (Fisher-Yates with crypto.getRandomValues) before passing to the next peer. Nobody — not even the assembler — can map inputs to outputs.

CCSH Protocol

CCSH (Cash Chat Secure Hub) is the 00 Protocol messaging protocol. Version 2 uses dual-channel split-knowledge encryption: each message is encrypted, XOR-split into two shares, then delivered via two independent networks — BCH OP_RETURN (on-chain, permanent) and Nostr relays (off-chain, ephemeral 1h TTL). Neither channel alone reveals anything.

Information-theoretic security — The XOR one-time pad is provably unbreakable (Shannon 1949). Even with unlimited computing power, a single share is indistinguishable from random noise. You need both the chain half and the relay half to recover a message.

Key Import

When you import a wallet, you provide either a BIP39 mnemonic (12 or 24 words) or a raw 128-character hex seed. Here is exactly what happens, step by step:

  • 01
    Input validation — The app checks that a seed is provided, the password is 8+ characters, and both password fields match.
  • 02
    Seed detection — If the input is a 128-char hex string, it's used directly as the 64-byte seed. Otherwise, it's treated as a BIP39 mnemonic phrase.
  • 03
    BIP39 seed derivation — The mnemonic is normalized (lowercase, single-spaced), then passed through PBKDF2-SHA512 with salt "mnemonic" and 2,048 iterations to produce a 64-byte master seed. This is the BIP39 standard used by Ledger, Trezor, and all HD wallets.
  • 04
    Key derivation — The 64-byte seed is passed through BIP32 + BIP44 to derive your BCH private key (see Key Derivation).
  • 05
    Vault encryption — The derived keys are encrypted with your password using AES-256-GCM and stored in localStorage (see Vault Encryption).
  • 06
    Session creation — A 30-minute session token is created so you don't need to re-enter your password on every page navigation.

Vault Encryption

Your private keys are never stored in plaintext. They are encrypted with AES-256-GCM before being saved to localStorage. Even if someone copies your browser data, they cannot access your keys without your password.

  • 01
    Salt generation — 16 random bytes are generated for PBKDF2 salt (unique per encryption).
  • 02
    IV generation — 12 random bytes are generated for AES-GCM initialization vector (unique per encryption).
  • 03
    Key stretching — Your password is passed through PBKDF2-SHA256 with 200,000 iterations to derive a 256-bit AES key. This makes brute-force attacks extremely expensive — each password guess requires 200,000 hash operations.
  • 04
    Encryption — Your keys (seed, private key, account-level keys) are serialized to JSON, then encrypted with AES-256-GCM. GCM mode provides both confidentiality and authentication — any tampering is detected.
  • 05
    Storage — The encrypted blob is stored as 00wallet_vault in localStorage. It contains: version, hex-encoded salt, hex-encoded IV, and hex-encoded ciphertext. No plaintext key material is ever written to disk.

    Master key — The 64-byte seed is hashed with HMAC-SHA512(key="Bitcoin seed", data=seed). Left 32 bytes = master private key, right 32 bytes = master chain code.
  • 02
    BIP44 path — Hardened child derivation at m/44'/145'/0' (purpose 44 = BIP44, coin 145 = BCH, account 0). This creates the account-level node from which all addresses are derived.
  • 03
    BCH key — Chain 0 (external), index 0: m/44'/145'/0'/0/0 → your main BCH address.
  • 04
    Stealth keys — Chain 2, index 0+1: scan key + spend key for stealth addresses (see Stealth Addresses).
  • 05
    BTC key — Chain 3, index 0: derived for cross-chain swaps via THORChain.
  • 06
    ETH key — Chain 4, index 0: derived for cross-chain operations.
// Real code from wallet.html — BIP32 derivation function bip32Master(seed) { const I = hmac(sha512, utf8('Bitcoin seed'), seed); return { priv: I.slice(0,32), chain: I.slice(32) }; } function deriveAccountNode(seed64) { // m/44'/145'/0' — account-level node let n = bip32Master(seed64); n = _bip32Child(n.priv, n.chain, 0x8000002c, true); // 44' n = _bip32Child(n.priv, n.chain, 0x80000091, true); // 145' (BCH) n = _bip32Child(n.priv, n.chain, 0x80000000, true); // 0' (account) return n; } function deriveBchPriv(seed64) { // m/44'/145'/0'/0/0 const acct = deriveAccountNode(seed64); let n = _bip32Child(acct.priv, acct.chain, 0, false); // external n = _bip32Child(n.priv, n.chain, 0, false); // index 0 return { priv: n.priv, acctPriv: acct.priv, acctChain: acct.chain }; }
One seed, all your keys. BCH, BTC, ETH, and stealth addresses are all mathematically derived from the same master seed. Back up one seed phrase → recover everything.

Session & Unlock

When you unlock your wallet (or import a new one), a session token is created so you don't have to re-enter your password every time you navigate between pages.

  • 01
    Token creation — After successful decryption, the app stores a session token in localStorage with key 00_session_auth. It contains the base64-encoded password and a timestamp.
  • 02
    30-minute TTL — The session expires after 30 minutes of inactivity. Each page load checks the timestamp: if more than 30 minutes have passed, you must unlock again.
  • 03
    Auto-refresh — When a page loads and the session is still valid, it refreshes the TTL so you stay logged in while actively using the app.
  • 04
    Cross-page sharing — The session works across all 00 Protocol pages (Wallet, Pay, Chat, DEX) via localStorage, which is shared across the same origin.

Stealth Addresses

Stealth addresses provide maximum privacy for receiving payments. Each payment creates a unique one-time address that only the recipient can detect and spend from. Even if someone knows your public stealth key, they cannot link incoming payments to each other.

  • 01
    Key pair — Two keys are derived from your seed: a scan key (m/44'/145'/0'/2'/0) and a spend key (m/44'/145'/0'/2'/1). The scan public key is shared; the spend private key never leaves your device.
  • 02
    ECDH shared secret — The sender computes a shared secret using their private key and the recipient's scan public key: shared = ECDH(sender_priv, recipient_scan_pub).
  • 03
    Tweak derivation — A tweak scalar is computed: c = SHA256(SHA256(shared_x) || outpoint). This is unique for every transaction.
  • 04
    One-time address — The stealth public key is computed by point addition on the elliptic curve: P = B_spend + c*G. This creates a completely new address that only the owner of the spend key can derive the private key for.
  • 05
    Spending — The recipient derives the spending key: priv = b_spend + c (scalar addition mod n). Only the recipient, who knows b_spend, can compute this.
// Real code from wallet.html — stealth address derivation function stealthDerive(senderPriv, senderPub, recipScanPub, recipSpendPub, outpointBytes) { // ECDH shared secret const sharedPoint = senderPriv ? secp256k1.getSharedSecret(senderPriv, recipScanPub) // sender : secp256k1.getSharedSecret(_stealthScanPriv, senderPub); // receiver const sharedX = sharedPoint.slice(1, 33); // Unique tweak per transaction const c = sha256(concat(sha256(sharedX), outpointBytes)); const cBig = BigInt('0x' + b2h(c)) % N_SECP; // P = B_spend + c*G (elliptic curve point addition) const spendPoint = secp256k1.ProjectivePoint.fromHex(recipSpendPub); const tweakPoint = secp256k1.ProjectivePoint.BASE.multiply(cBig); const stealthPoint = spendPoint.add(tweakPoint); return { addr: stealthAddr, pub: stealthPub, cBig }; } // Spending key: priv = b_spend + c (mod n) function stealthSpendingKey(spendPriv, cBig) { const bBig = BigInt('0x' + b2h(spendPriv)); const pBig = (bBig + cBig) % N_SECP; return h2b(pBig.toString(16).padStart(64, '0')); }
PRIVACY Stealth addresses break on-chain link analysis. A blockchain observer sees a random one-time address — they cannot link it to the recipient's main wallet or to any other stealth payment.
