1. Overview

00 Protocol implements beaconless stealth addresses on Bitcoin Cash using ECDH (Elliptic Curve Diffie-Hellman) on secp256k1. No on-chain beacon (OP_RETURN) is needed. The sender derives a one-time address from the receiver's public stealth paycode, and notifies the receiver via an encrypted Nostr DM.

Properties:
  • Unlinkable — each payment creates a unique address with no on-chain link to the receiver
  • Beaconless — no OP_RETURN or notification transaction on-chain
  • Wallet-agnostic ECDH — stealthDerive() takes raw keys, agnostic to derivation source
  • Multi-wallet support — works with both BIP39 seed phrases AND raw hex keys
  • Cross-wallet tested — 12-word wallet can send to raw-key wallet and vice versa
  • Ephemeral sender identity — Nostr DM sent with a random key, unlinkable to sender wallet
Receiver shares paycode -> Sender derives one-time addr -> Sender broadcasts TX -> Sender sends Nostr DM -> Receiver detects payment

2. Key Derivation

The stealth system requires two key pairs: scan (for detecting payments) and spend (for spending received funds). These are derived differently depending on the wallet type.

Path A: BIP39 Mnemonic (12/24 words) Recommended

BIP352-style key derivation. Stealth keys live in a separate tree from the BIP44 wallet. Scan and spend are behind independent hardened gates — compromising the scan branch gives zero access to the spend branch.

BIP352 stealth tree (separate from BIP44):
m/352'/145'/0'
  +-- 0'  spend  (HARDENED)           <- isolated from scan
  |    +-- 0  spend key (non-hardened)
  +-- 1'  scan   (HARDENED)           <- isolated from spend
       +-- 0  scan key  (non-hardened)

Full paths:
  spend: m/352'/145'/0'/0'/0
  scan:  m/352'/145'/0'/1'/0
// Implementation (wallet.html, lines 7204-7218)
const bip352     = deriveBip352Node(seed64);                              // m/352'/145'/0'
const spendBranch = _bip32Child(bip352.priv, bip352.chain, 0, true);     // /0' (HARDENED)
const scanBranch  = _bip32Child(bip352.priv, bip352.chain, 1, true);     // /1' (HARDENED)
const spendChild  = _bip32Child(spendBranch.priv, spendBranch.chain, 0, false);  // /0
const scanChild   = _bip32Child(scanBranch.priv, scanBranch.chain, 0, false);    // /0

_stealthSpendPriv = spendChild.priv;
_stealthSpendPub  = getPublicKey(spendChild.priv, true);
_stealthScanPriv  = scanChild.priv;
_stealthScanPub   = getPublicKey(scanChild.priv, true);
_stealthNostrPub  = b2h(_stealthScanPub.slice(1));  // x-only for Nostr
Design Decision (BIP352): Scan and spend live in separate hardened branches under m/352'/145'/0'. If the scan xpub leaks, the attacker can detect stealth payments but cannot derive the spend key. If the spend xpub leaks, the attacker knows the spend pubkey but cannot scan for payments. This enables safe watch-only scanning and WizardConnect integration (wallet sends scan xpub to dapp).
Recovery: Same seed phrase -> same stealth keys -> same paycode. Deterministic. Recoverable from backup.

Path B: Raw Private Key (hex) Tested & Working

When no BIP32 chain code is available (raw key import), stealth keys are derived via SHA256 with domain separation (inspired by Umbra/ERC-5564):

// Implementation (wallet.html, lines 7215-7229)
const enc = new TextEncoder();
_stealthScanPriv  = sha256(concat(enc.encode('bch-stealth-scan'), _bchPriv32));
_stealthSpendPriv = sha256(concat(enc.encode('bch-stealth-spend'), _bchPriv32));
_stealthScanPub   = secp256k1.getPublicKey(_stealthScanPriv, true);
_stealthSpendPub  = secp256k1.getPublicKey(_stealthSpendPriv, true);
_stealthNostrPub  = b2h(_stealthScanPub.slice(1));
Deterministic: Same raw key -> same SHA256 -> same stealth keys -> same paycode. Verified: disconnect/reconnect produces identical paycode.
Important: The paycode generated from a raw key is different from the paycode generated from the same seed via BIP32. A user cannot switch between methods and keep the same paycode. Cross-wallet stealth (12-word to raw-key) is tested and works because the ECDH is key-agnostic.

4. ECDH Derivation (stealthDerive)

The core derivation function. Takes raw keys as input — completely agnostic to the key source (BIP32, SHA256, hardware wallet, etc.).

// wallet.html, line 4129
function stealthDerive(senderPriv, senderPub, recipScanPub, recipSpendPub, outpointBytes) {

  // Step 1: ECDH shared secret
  const sharedPoint = senderPriv
    ? secp256k1.getSharedSecret(senderPriv, recipScanPub)   // SENDER side
    : secp256k1.getSharedSecret(_stealthScanPriv, senderPub); // RECEIVER side

  // Step 2: Extract x-coordinate (drop the prefix byte)
  const sharedX = sharedPoint.slice(1, 33);  // 32 bytes

  // Step 3: Derive tweak scalar c
  const c = sha256(concat(sha256(sharedX), outpointBytes));
  const cBig = BigInt('0x' + b2h(c)) % N_SECP;

  // Step 4: Compute stealth public key: P = B_spend + c*G
  const spendPoint = secp256k1.ProjectivePoint.fromHex(recipSpendPub);
  const tweakPoint = secp256k1.ProjectivePoint.BASE.multiply(cBig);
  const stealthPoint = spendPoint.add(tweakPoint);
  const stealthPub = stealthPoint.toRawBytes(true);  // 33 bytes compressed

  // Step 5: Convert to BCH CashAddr
  const stealthAddr = pubHashToCashAddr(Array.from(ripemd160(sha256(stealthPub))));

  return { addr: stealthAddr, pub: stealthPub, cBig };
}

ECDH Symmetry Proof

Sender:   ECDH(senderPriv, recipScanPub) = senderPriv * scanPub
        = senderPriv * (scanPriv * G) = senderPriv * scanPriv * G

Receiver: ECDH(scanPriv, senderPub)     = scanPriv * senderPub
        = scanPriv * (senderPriv * G)   = scanPriv * senderPriv * G

Both produce the SAME point: senderPriv * scanPriv * G  (verified in tests)

Tweak Derivation

c = SHA256( SHA256(sharedX) || outpointBytes )

Where:
  sharedX      = x-coordinate of ECDH shared point (32 bytes)
  outpointBytes = prevTxID_LE (32 bytes) || prevVout_LE (4 bytes)
  ||           = byte concatenation

The double-hash (SHA256 of SHA256(sharedX)) provides domain separation.

Spending Key Derivation

// wallet.html, line 4145
function stealthSpendingKey(spendPriv, cBig) {
  const bBig = BigInt('0x' + b2h(spendPriv));
  const pBig = (bBig + cBig) % N_SECP;         // p = b + c mod n
  return h2b(pBig.toString(16).padStart(64, '0'));
}

// Verification: p*G = (b+c)*G = b*G + c*G = B_spend + c*G = stealthPub

5. Send Flow (Sender Side)

File: wallet.html, function sendStealth(), line 4307

  1. Parse paycode — extract recipScanPub (33 bytes) and recipSpendPub (33 bytes), validate as secp256k1 points
  2. Validate — amount >= 546 sats (dust limit)
  3. Select UTXOs — greedy selection from regular UTXOs only (stealth UTXOs excluded via stealthAddrSet filter)
  4. Get sender key — use first selected UTXO's private key for ECDH: firstUtxo._priv || _bchPriv32
  5. Construct outpoint — h2b(firstUtxo.txid).reverse() || u32LE(firstUtxo.vout)
  6. Derive one-time address — stealthDerive(senderPriv, null, recipScanPub, recipSpendPub, outpoint)
  7. Build transaction — P2PKH output to stealth address + change output, sign with ECDSA DER + sighash 0x41 (FORKID)
  8. Broadcast — via Fulcrum blockchain.transaction.broadcast
  9. Notify via Nostr — encrypted DM with {type:'stealth', txid} using ephemeral key
// Step 4-6 (lines 4354-4361)
const senderPriv = firstUtxo._priv || _bchPriv32;
const outpoint = concat(h2b(firstUtxo.txid).reverse(), u32LE(firstUtxo.vout));
const { addr, pub } = stealthDerive(senderPriv, null, recipScanPub, recipSpendPub, outpoint);

// Step 9: Nostr DM (lines 4395-4400)
const ephPriv = rand(32);                                    // random key (unlinkable)
const recipNostrPub = b2h(recipScanPub.slice(1));            // x-only (32 bytes)
const payload = JSON.stringify({ type: 'stealth', txid });
const encrypted = await nip04Encrypt(ephPriv, recipNostrPub, payload);
const dmEvent = await makeNostrEvent(ephPriv, 4, encrypted, [['p', recipNostrPub]]);
nostrPublish(dmEvent);
Input ordering: The ECDH uses the first selected UTXO's key. UTXOs are sorted by value (descending), so the largest UTXO becomes the first input. The receiver's handleStealthEvent tests ALL input pubkeys for robustness.

6. Receive Flow (Receiver Side)

The receiver uses the sender's input public key (extracted from the transaction's scriptSig) and their own scan private key to compute the same ECDH shared secret.

Detection Algorithm (handleStealthEvent, line 4213)

For each incoming Nostr DM (kind 4):
  1. Decrypt NIP-04 content with scanPriv
  2. Parse JSON: extract txid
  3. Fetch raw TX hex via Fulcrum
  4. Extract ALL input pubkeys (parseTxAllInputPubkeys)
  5. For each input pubkey:
     a. Compute ECDH(scanPriv, inputPubkey) -> sharedSecret
     b. Extract outpoint (prevTxID_LE || prevVout_LE)
     c. Compute tweak: c = SHA256(SHA256(sharedX) || outpoint)
     d. Derive stealth pubkey: P = spendPub + c*G
     e. Compute P2PKH script from hash160(P)
     f. Compare with each output script
     g. If match -> STEALTH PAYMENT DETECTED
        - Compute spending key: p = spendPriv + c mod n
        - Store {txid, vout, value, priv, pub, addr, from, ts}

7. Scanning Methods

MethodTriggerScopeSpeed
Nostr DM Automatic (real-time) Specific TX (sender notifies) Instant (1-3 sec)
Periodic Auto-Scan Every 60 seconds New TXs on registered stealth addresses Fast
Manual Scan Chain User clicks "Scan Chain" All TXs in own address history (incremental) Medium
Advanced Deep Scan User selects date range ALL TXs in ALL blocks in range Slow (via Blockchair API)
Fundamental limitation of Scan Chain / Periodic Scan: These only scan transactions touching the receiver's KNOWN addresses. A stealth payment from a stranger goes to a one-time address not yet known. These methods work AFTER initial detection (Nostr DM or Advanced Scan) registers the stealth address.

For true beaconless detection without Nostr: The Advanced Deep Scan must be used — it scans ALL transactions in the specified block range, testing every input pubkey via ECDH.

8. Spending Stealth UTXOs

Stealth UTXOs are registered in _hdAddresses via _registerStealthAddresses() (line 4287) with their tweaked private keys, enabling the normal send flow to spend them.

// _registerStealthAddresses (line 4287)
function _registerStealthAddresses() {
  for (const su of _stealthUtxos) {
    if (_hdAddresses.some(a => a.addr === su.addr)) continue;
    _hdAddresses.push({
      priv: h2b(su.priv),        // tweaked spending key: spendPriv + c
      pubKey: h2b(su.pub),       // tweaked pubkey: spendPub + c*G
      addr: su.addr,             // one-time stealth address
      path: 'stealth',
    });
  }
}

10. Security Considerations

Known Limitations

  • Nostr relay dependency: Instant detection requires at least one common relay between sender and receiver. If all relays are down, the receiver must use Advanced Scan.
  • Raw key paycode differs from BIP32: Same entropy produces different stealth keys depending on whether it's imported as mnemonic or raw key.
  • Non-hardened derivation at /2: If the account xpub leaks, stealth public keys can be derived (but not private keys). Deliberate trade-off for WizardConnect/watch-only support.
  • Periodic/Chain scan limitation: Only scans known addresses. Cannot detect stealth payments from unknown senders without Nostr DM or Advanced Scan.

Privacy Properties

  • On-chain: stealth outputs are indistinguishable from regular P2PKH outputs
  • Sender unlinkability: Nostr DM uses ephemeral key, cannot link DM to sender wallet
  • Plausible deniability: without the scan key, stealth UTXOs are undetectable. Wallet appears to have 0 BCH
  • Nostr metadata: relay operators can see the x-only scan pubkey (recipient), but not the content
